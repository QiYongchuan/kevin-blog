# 复利工程文档：GitHub Issues + Cloudflare 快速建站避坑指南

> 版本：v1.0 | 创建：2025-04-09 | 适用：Next.js + GitHub Issues + Cloudflare Pages 建站

---

## 一、项目架构速查

```
GitHub Issues (CMS) → Next.js (SSG) → Cloudflare Pages (CDN)
         ↓                    ↓               ↓
    写内容               构建静态页         全球访问
    打标签               ISR 增量更新       自动 HTTPS
```

**核心优势**：零成本、零维护、随处可写、10 秒发布

## 二、常见大坑与解决方案

### 🚨 坑 1：Cloudflare Pages 分支陷阱

**现象**：
- 本地修改了代码，wrangler 部署成功
- 但访问自定义域名（如 imduoduo.com）还是旧版本
- 预览链接（xxx.pages.dev）却是最新的

**原因**：
Cloudflare Pages 的 Production 部署绑定特定分支（通常是 `master` 或 `main`），推送到其他分支会变成 Preview 部署，不影响正式域名。

**解决**：
```bash
# 先检查项目绑定的是哪个分支
wrangler pages deployment list --project-name your-project

# 部署时必须匹配 Production 分支
wrangler pages deploy dist --project-name your-project --branch master
# 或
wrangler pages deploy dist --project-name your-project --branch main
```

**预防**：
- 在项目设置中确认 Production 分支
- 团队约定统一使用 `main` 或 `master`

---

### 🚨 坑 2：Next.js 图片优化与静态导出冲突

**现象**：
```
Error: Image Optimization using Next.js' default loader is not compatible with `output: 'export'`.
```

**解决**：
必须关闭图片优化：
```javascript
// next.config.mjs
const nextConfig = {
  output: 'export',
  distDir: 'dist',
  images: {
    unoptimized: true,  // ← 必须加这个
  },
};
```

**替代方案**：
- 使用外部图床（Cloudflare Images、Imgur、GitHub Assets）
- 使用 `<img>` 标签代替 Next.js `<Image>`

---

### 🚨 坑 3：GitHub API 速率限制

**现象**：
- 本地开发时突然获取不到 Issues
- 控制台报错 `403 API rate limit exceeded`

**原因**：
- 无 Token：每小时 60 次请求
- 有 Token：每小时 5000 次请求

**解决**：
1. 创建 GitHub Token：https://github.com/settings/tokens
2. 只需要 `public_repo` 权限（读取公开仓库）
3. 环境变量配置：
   ```
   # .env.local
   NEXT_PUBLIC_GITHUB_TOKEN=ghp_xxxxxxxx
   ```

**代码中正确使用**：
```typescript
const GITHUB_TOKEN = process.env.NEXT_PUBLIC_GITHUB_TOKEN || '';

async function fetchGitHubAPI(url: string) {
  const headers: HeadersInit = {
    'Accept': 'application/vnd.github.v3+json',
  };

  if (GITHUB_TOKEN) {
    headers['Authorization'] = `Bearer ${GITHUB_TOKEN}`;
  }

  const response = await fetch(url, { headers });
  // ...
}
```

### 🚨 坑 4：GitHub Actions 触发时机不对

**现象**：
- 发布了 Issue，但博客没有自动更新
- 手动触发 Action 才生效

**原因**：
GitHub Actions 的 `on: issues` 事件类型不完整。

**正确配置**：
```yaml
# .github/workflows/deploy.yml
on:
  issues:
    types: [opened, edited, deleted, closed, reopened, labeled, unlabeled]
  workflow_dispatch:  # 支持手动触发
```

**注意**：
- `edited`：修改 Issue 内容时触发
- `labeled/unlabeled`：添加/移除标签时触发
- 如果只配 `opened`，修改文章不会触发重建

### 🚨 坑 5：标签过滤逻辑踩坑

**现象**：
- 打了 `hidden` 标签的文章还是显示出来了
- 或者打了标签的文章不显示

**常见错误代码**：
```typescript
// ❌ 错误：没有处理 label 可能是对象的情况
const labels = issue.labels; // 可能是 string[] 或 {name: string}[]
```

**正确处理**：
```typescript
// ✅ 正确：统一处理 label 格式
const labels = issue.labels?.map((label: string | GitHubLabel) =>
  typeof label === 'string' ? label : label.name
) || [];

// 过滤逻辑
const isHidden = labels.some((l: string) => 
  l.toLowerCase() === 'hide' || l.toLowerCase() === 'hidden'
);

const isThought = labels.some((l: string) => 
  l.toLowerCase() === 'thought'
);
```

**标签设计建议**：
| 标签 | 用途 | 显示位置 |
|------|------|---------|
| 无/任意 | 普通文章 | Home |
| `thought` | 碎片想法 | Thoughts 页面 |
| `hidden` | 草稿/隐藏 | 不显示 |

### 🚨 坑 6：ISR 与静态导出的误解

**现象**：
```typescript
// 在 layout.tsx 或 page.tsx 中设置
export const revalidate = 600;  // 10 分钟
```
以为这会让网站每 10 分钟自动更新。

**真相**：
- ISR (Incremental Static Regeneration) 需要服务器支持
- `output: 'export'` 是纯静态导出，**不支持 ISR**
- `revalidate` 在静态导出中会被忽略

**真正的自动更新方案**：
1. **GitHub Actions**：Issue 变动时触发构建（推荐）
2. **定时触发**：GitHub Actions 定时任务
   ```yaml
   on:
     schedule:
       - cron: '0 */6 * * *'  # 每 6 小时构建一次
   ```
3. **Cloudflare Workers**：边缘函数定时刷新

### 🚨 坑 7：动态路由的静态生成参数

**现象**：
- `npm run build` 时报错
- `Error: generateStaticParams() must be used with output: 'export'`

**正确做法**：
```typescript
// app/post/[id]/page.tsx
export async function generateStaticParams() {
  // 构建时获取所有文章 ID
  const posts = await getAllPosts();
  
  return posts.map((post) => ({
    id: post.id.toString(),
  }));
}
```

**注意**：
- 必须返回数组，每个元素包含路由参数
- 参数值必须是字符串
- 如果构建时 API 失败，构建会失败（需要加 fallback 处理）

### 🚨 坑 8：环境变量前缀问题

**现象**：
- 本地 `.env.local` 配置了 Token，开发正常
- 部署到 Cloudflare 后 Token 不生效

**原因**：
- Next.js 中 `process.env.Xxx` 只在服务端可用
- 浏览器端需要 `NEXT_PUBLIC_` 前缀

**解决方案**：
```
# .env.local
NEXT_PUBLIC_GITHUB_TOKEN=ghp_xxxxxxxx
```

```typescript
// 代码中使用
const token = process.env.NEXT_PUBLIC_GITHUB_TOKEN;
```

**风险提示**：
- `NEXT_PUBLIC_` 会暴露给客户端
- 只用于公开只读 Token，不要放有写权限的 Token

### 🚨 坑 9：Cloudflare Pages 项目名与仓库名不同

**现象**：
```
[ERROR] Project not found. The specified project name does not match any of your existing projects.
```

**解决**：
```bash
# 先列出所有项目
wrangler pages project list

# 使用正确的项目名部署
wrangler pages deploy dist --project-name 正确的项目名
```

### 🚨 坑 10：Markdown 渲染问题

**现象**：
- Markdown 中的 HTML 标签不渲染
- 代码块样式不对
- 链接不跳转

**推荐方案**：
```bash
npm install react-markdown remark-gfm rehype-raw prism-react-renderer
```

```typescript
import ReactMarkdown from 'react-markdown';
import remarkGfm from 'remark-gfm';
import rehypeRaw from 'rehype-raw';

<ReactMarkdown
  remarkPlugins={[remarkGfm]}  // 支持表格、删除线等 GitHub 风格
  rehypePlugins={[rehypeRaw]}   // 支持 HTML 标签
  components={{
    // 自定义代码块样式
    code({ node, inline, className, children, ...props }) {
      // ... 实现代码高亮
    }
  }}
>
  {content}
</ReactMarkdown>
```

## 三、快速建站 SOP

### Step 1：创建 GitHub 仓库（2 分钟）
- 新建公开仓库
- 记录用户名和仓库名

### Step 2：本地初始化项目（5 分钟）
```bash
npx create-next-app@14 my-blog --typescript --tailwind --eslint --app --no-src-dir
cd my-blog
```

### Step 3：配置静态导出（3 分钟）
修改 `next.config.mjs`：
```javascript
const nextConfig = {
  output: 'export',
  distDir: 'dist',
  images: { unoptimized: true },
};
export default nextConfig;
```

### Step 4：创建 API 层（10 分钟）
复制 `lib/api.ts`，修改：
- `REPO_OWNER = '你的用户名'`
- `REPO_NAME = '你的仓库名'`

### Step 5：创建页面（15 分钟）
- `app/page.tsx` - 首页文章列表
- `app/post/[id]/page.tsx` - 文章详情
- `app/layout.tsx` - 导航和布局

### Step 6：部署到 Cloudflare（10 分钟）
```bash
# 安装 wrangler
npm install -g wrangler

# 登录
wrangler login

# 创建项目
wrangler pages project create my-blog

# 构建
npm run build

# 部署（注意分支！）
wrangler pages deploy dist --project-name my-blog --branch master
```

### Step 7：配置自动化（5 分钟）
创建 `.github/workflows/deploy.yml`，添加：
- `CLOUDFLARE_API_TOKEN` 到 GitHub Secrets

**总耗时：约 50 分钟（熟练后可压缩到 20 分钟）**

## 六、经验教训总结

### 做得好的
1. **GitHub Issues 作为 CMS** - 极大的降低了写作 friction
2. **纯静态导出** - 零服务器成本，极快访问速度
3. **标签驱动内容** - 灵活的分类系统，无需修改代码

### 踩过的坑
1. 初期没有正确处理 GitHub API 的 label 格式
2. 部署时分支不匹配导致正式域名不更新
3. 图片优化配置遗漏导致构建失败

### 关键决策
1. 选择 Cloudflare Pages 而非 Vercel（免费额度更友好）
2. 使用 ISR revalidate 作为缓存策略提示（虽然静态导出不真正支持）
3. 分离 Home（文章）和 Thoughts（碎片）两个入口
